HTTP Versions

From the early web to modern high-performance protocols.

Version Overview

Version Year Key Features
HTTP/0.9 1991 Simple text retrieval only; no headers or status codes
HTTP/1.0 1996 Added headers, status codes, and content types
HTTP/1.1 1997 Persistent connections, better caching, virtual hosting
HTTP/2 2015 Multiplexing, header compression, binary framing
HTTP/3 2022 Runs on QUIC, faster connection setup, better on mobile

HTTP/0.9 - The Beginning

The history of HTTP begins in the early 1990s when Tim Berners-Lee developed the first version while working at CERN. HTTP/0.9 supported only simple text retrieval and lacked headers or status codes. It assumed a very limited use case: scientists sharing static documents over trusted networks. Performance, security, and scalability were not major concerns because the web itself was small and experimental.

HTTP/1.0 - Expanding Capabilities

HTTP/1.0 introduced headers, status codes, and content types, which allowed browsers to display images and different file formats. These additions separated metadata from content and allowed clients and servers to negotiate how information should be handled.

However, HTTP/1.0 still opened a new TCP connection for every request, which caused inefficiency as webpages became more complex.

HTTP/1.1 - The Backbone of the Web

HTTP/1.1 improved performance by adding:

  • Persistent connections - multiple requests reuse a single TCP connection
  • Better caching - browsers can reuse stored resources to reduce data transfers
  • Virtual hosting - multiple websites can share a single IP address

This version became the backbone of the web for more than a decade and still remains widely supported today due to its stability and compatibility.

HTTP/2 - Performance Improvements

As the web expanded, HTTP/1.1 began to show limits. Modern webpages load many resources including scripts, images, fonts, and media files, sometimes requiring dozens or hundreds of requests. HTTP/2 addressed these issues by introducing:

  • Multiplexing - multiple requests travel simultaneously over one connection
  • Header compression - reduces repetitive metadata and bandwidth usage
  • Binary framing - machines process data faster than text-based formatting

These improvements allowed the protocol to perform better without changing how developers wrote applications.

HTTP/3 - The Next Step

HTTP/3 runs on the QUIC transport protocol instead of TCP, reducing connection setup time and improving performance on unreliable networks. QUIC handles packet loss more efficiently and avoids head-of-line blocking, a common issue with TCP-based connections.

This benefits users on mobile, wireless, or congested networks. Major companies and content delivery networks already support HTTP/3, and adoption continues to grow.
