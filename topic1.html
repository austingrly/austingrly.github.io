HTTP Basics

Requests, Responses, and the Structure of Web Communication

What Is HTTP?

HTTP was made to solve the problem of systems being able to exchange web resources over a network. When you type an address into the browser, that browser sends an HTTP request to the server. The server processes that request and sends back an HTTP response. This exchange happens continuously in the background, often hundreds of times when loading a single page.

HTTP works on top of lower-level protocols such as TCP/IP, which handles routing and delivery. HTTP defines what the message represents, what action will occur, and how the data should be interpreted when received.

Structure of an HTTP Transaction

Every HTTP transaction includes the following components:

  • Request - tells the server what the client is looking for
  • Response - tells the client what occurred
  • Headers - provide extra info (content type, caching, authentication)
  • Body - carries the actual data being transferred

HTTP Request Methods

Every request includes a method that specifies what action should be performed. The four most common methods are:

Method Purpose Example Use
GET Retrieve data from the server Loading a webpage or image
POST Submit new data to the server Submitting a form or login
PUT Update existing data on the server Editing a profile or record
DELETE Remove data from the server Deleting an account or post

HTTP Status Codes

The response includes a status code that tells the client what occurred. This allows browsers and apps to react appropriately without guessing the situation.

  • 200 OK - The request was successful
  • 301 Moved Permanently - The resource has been moved to a new URL
  • 400 Bad Request - The server could not understand the request
  • 401 Unauthorized - Authentication is required
  • 403 Forbidden - The server refuses to fulfill the request
  • 404 Not Found - The requested resource does not exist
  • 500 Internal Server Error - The server encountered an unexpected condition

HTTP and Security: HTTPS

HTTP itself does not encrypt data, which allowed attackers to intercept traffic, steal credentials, and manipulate content. HTTPS solved this problem by combining HTTP with Transport Layer Security (TLS). HTTPS encrypts data in transit and verifies server identity, protecting users from eavesdropping and tampering.

Modern browsers now warn users when websites do not use HTTPS, and many web features require it by default.
